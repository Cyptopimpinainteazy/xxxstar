# Post-Task Documentation Update Prompt

After any task is completed, run a documentation synchronization pass to keep all related markdown files accurate, consistent, and traceable.

## Trigger
Run this workflow immediately after task completion when the task changes code, tests, architecture, runtime behavior, deployment steps, status, or readiness.

## Required Inputs
- Task name
- Test results
- Build logs
- Timestamp in UTC
- Key metrics (coverage, pass/fail counts, durations, error counts)

## Workflow
1. Identify related documentation files.
- Scan all markdown files and include docs that mention the touched module, feature, phase, or subsystem.
- Include status files, readiness reports, phase summaries, QA docs, indexes, and operation guides.

2. Classify update priority.
- High: direct task references and status/readiness files.
- Medium: phase docs, QA docs, and execution playbooks.
- Low: index, navigation, and background docs.

3. Update content with evidence.
- Add completed status with timestamp.
- Add test results and build outcome.
- Add metrics and acceptance criteria evidence.
- Add risks, gaps, and next action.

4. Validate cross-file consistency.
- Ensure dates and status labels match.
- Ensure links and references are valid.
- Ensure no file contradicts current state.

5. Produce a session summary.
- Include what changed, files modified, evidence used, and remaining gaps.

## Output Format
Use this concise completion block:

```text
DOCUMENTATION UPDATE REPORT
Task: <task name>
Timestamp: <YYYY-MM-DD HH:MM:SS UTC>
Test Results: <summary>
Build Result: <summary>
Metrics: <key metrics>
Files Updated: <count and list>
Consistency Check: PASS/FAIL
Next Action: <one concrete next step>
```

## Quality Rules
- Do not claim completion without evidence.
- If evidence is missing, mark status as partial and list blockers.
- Prefer under-claiming over over-claiming.
- Keep updates factual and operational.
