# Design Notes — add-swarm-orchestra-platform

This change turns the swarm-orchestra idea into a bounded platform model anchored to the code that already exists. The intent is to keep the system modular enough for specialized execution while preventing role confusion, silent privilege growth, and undocumented side channels between services.

The first production slice has four concrete foundations. `x3-gpu-validator-swarm` owns deterministic GPU validation, CPU verification, replay, quarantine, and fallback. `x3-sidecar` owns benchmark execution, receipt generation, and report publication plumbing. `x3-gateway` owns external REST and GraphQL entry points. The runtime and node continue to own governance, consensus, and validator admission. The earlier draft implied that the older `gpu-swarm` and `pallet-swarm` path was the place to expand the platform. That is no longer true for a production rollout because both are explicitly deprecated.

The platform uses a conductor pattern rather than a hive-mind pattern. The conductor accepts intents, routes work to role-typed services, enforces policy, and records resulting evidence. Agents do not gain legitimacy merely because they can do work. They remain legitimate only when they stay inside the approval, security, and audit path.

The orchestra-control-plane is a new off-chain service boundary, not a runtime pallet and not a validator extension. It should own five tables or aggregates from the start: intents, approval cases, vote windows, evidence bundles, and reward accruals. Gateway routes should expose write APIs for intents, approvals, and votes, and read APIs for benchmark reports, workflow status, and evidence history. Sidecar should stay narrow: it executes benchmarks, tracks local job state, and publishes signed benchmark artifacts. It should not absorb CRM logic, approval logic, or outward-content logic.

Compute routing is part of the design, not an implementation afterthought. GPU-heavy work such as inference, graph analytics, backtesting, and benchmark execution should be schedulable across local and burst capacity, while low-latency control services remain CPU-oriented. The control plane therefore needs metadata about sensitivity, latency budget, determinism class, approval class, and cost ceiling before assigning work.

Two mismatches had to be closed before this design could be treated as a valid rollout base. First, the startup determinism gate existed in the runtime but was not enforced by the authority node path. Second, governance periods in the runtime were still sized for an obsolete 6-second block assumption even though the runtime targets 200ms blocks. This change treats both as blocking safety prerequisites rather than documentation nits.

The security swarm is not a sidecar. It is the enforcement substrate for the entire platform. Every financially meaningful execution path, externally visible publication path, and exceptional infrastructure action should emit lineage that the security and evidence layers can consume. `x3-court` remains the deterministic replay-and-slashing court for protocol disputes; it does not become the human approval board or CRM voting engine.
